클로드 코드(Claude Code) 실전 개발 워크플로우 분석 요약

💡 가장 핵심적인 대원칙: 기획과 코딩의 철저한 분리

이 워크플로우를 관통하는 단 하나의 원칙은 "작성된 계획을 개발자가 직접 검토하고 승인하기 전까지 AI가 절대 코드를 쓰지 못하게 하는 것"입니다. 이를 통해 무의미한 삽질을 줄이고, 아키텍처의 주도권을 개발자가 유지하며, 토큰 낭비 없이 훨씬 더 나은 결과물을 도출할 수 있습니다.

1. 리서치 (Research): 실패를 막는 심층 분석

  • 문서화 필수: 작업을 시작할 때 AI에게 관련 코드를 깊이 읽고 분석하도록 지시하며, 결과물은 채팅창 요약이 아닌 반드시 research.md라는 별도의 마크다운 파일로 작성하게 해야 합니다.
  • 강력한 프롬프트 사용: 지시를 내릴 때 '깊이', '매우 상세히', '모든 세부 사항' 같은 단어를 반드시 포함해야만 AI가 코드를 대충 훑어보는 것을 막을 수 있습니다.
  • 비용이 큰 실패 방지: 이 단계는 문법 에러가 아니라, 기존 레이어를 무시하거나 중복된 로직을 생성하여 주변 시스템을 조용히 망가뜨리는 '가장 치명적인 구현 실패'를 사전에 차단합니다.

2. 기획 (Planning): 상세 구현 설계도 작성

  • 상세 플랜 문서화: 리서치가 끝나면 어떻게 구현할지에 대한 접근 방식, 코드 스니펫, 수정 경로 등을 담은 plan.md 파일을 작성하게 합니다.
  • 내장 기능 대신 MD 파일 사용: AI의 내장 플랜 모드를 쓰지 않고 실제 마크다운 파일을 사용하는 이유는 에디터에서 직접 열어 인라인 메모를 달기 쉽고, 영구적인 산출물로 남아 세션이나 기기가 변경되어도 작업의 연속성을 보장할 수 있기 때문입니다.

3. 검토 및 피드백 (Review & Feedback): 주도권을 쥐는 핵심 루프

  • 인라인 메모 작성 (Shared Mutable State 패턴): 채팅창에서 말로 설명하는 대신, 개발자가 직접 plan.md를 열고 수정이나 거부가 필요한 위치에 구체적인 제약 사항이나 비즈니스 로직에 대한 메모(주석)를 적습니다.
  • 가장 중요한 프롬프트: 메모 작성 후 AI에게 다시 넘길 때 "메모를 반영해서 문서를 업데이트해. 단, 아직 구현하지 마"라고 명시적으로 지시해야만 AI가 스스로 충분하다고 판단하여 멋대로 코드를 작성하는 것을 막을 수 있습니다.
  • 개발자가 계획에 100% 만족할 때까지 이 검토와 업데이트 과정을 반복합니다.

4. 실제 구현 및 감독 (Implementation): 기계적 실행

  • 기계적인 코딩: 창의적이고 중요한 설계 결정은 이전 단계에서 모두 끝났으므로, 실제 코드를 작성하는 과정은 철저히 기계적이고 지루한 실행 작업이 됩니다.
  • 감독자 역할 전환: 개발자는 "전부 구현해라, 완료 시 계획 문서에 표시해라, 타입 체크를 지속적으로 실행해라"와 같은 표준화된 지시를 내리는 감독자가 되며, UI/UX 수정에는 스크린샷을 적극 활용합니다.
  • Git을 활용한 오류 대처: AI가 코드를 잘못된 방향으로 작성했다면, 그 위에서 덧붙여 수정(패치)하려 하지 말고 Git을 이용해 통째로 되돌린(Reset/Revert) 후 작업 범위를 좁혀 다시 시도하는 것이 항상 더 나은 결과를 만듭니다.

5. 세션 및 주도권 관리 (Session & Control)

  • 단일 롱 세션 유지: 작업을 여러 세션으로 쪼개지 않고 하나의 긴 세션으로 유지합니다. 영구적인 MD 파일이 있고 AI의 자체 압축(Auto Compaction) 기능이 동작하기 때문에 컨텍스트 윈도우 한계로 인한 성능 저하를 방지할 수 있습니다.
  • 결정권은 100% 개발자에게: AI에게 실행은 맡기되, 무엇을 만들고 어떤 라이브러리를 쓸지 등 핵심 의사결정의 운전대는 절대 넘겨주지 않아야 합니다.
