# **보험 상품 AI 약관 분석 서비스**

## 

## **1\. 사업 개요**

서비스 운영자가 사전에 구축한 \*\*'보험 상품 및 약관 데이터베이스'\*\*를 기반으로 즉각적인 보장 분석 및 검색 서비스를 제공함. 높은 데이터 신뢰도와 끊김 없는 사용자 경험(Seamless UX)을 최우선 가치로 설정함.

## 

## **2\. 서비스 핵심 프로세스 (UX/UI)**

1. **상품 검색:** 사용자가 상품명, 보험사, 또는 출시 연도 키워드로 본인의 보험을 검색.  
2. **데이터 매칭:** 시스템 내에 미리 벡터화되어 저장된 해당 상품의 약관 데이터를 호출.  
3. **AI 분석/질의:** "이 보험에서 갑상선암은 얼마나 보상해줘?"와 같은 질문에 AI가 미리 학습된 약관을 근거로 즉시 답변.

## 

## **3\. 기술 구현 전략 (Technical Stack)**

### 

### **3.1. 관리자 선행 데이터 구축 (Admin-Side)**

* **Data Sourcing:** 각 보험사 상품 공시실 및 보험개발원의 공시 약관을 주기적으로 수집.  
* **데이터 구조화:** 추출된 텍스트를 문맥 단위(Chunk)로 분할하고, `보험사`, `상품군`, `출시일`, `개정일` 등의 메타데이터를 부여함.

### 

### **3.2. 벡터 데이터베이스 및 검색, 조회 엔진**

* **저장소:** **PostgreSQL** 기반의 **pgvector** 확장 모듈을 사용하여 임베딩된 약관 데이터를 저장.  
* **Vectorization Pipeline:** 수집된 PDF 데이터를 텍스트로 추출하여 \*\*PostgreSQL(pgvector)\*\*에 저장.  
  * *Tip:* 상품명, 보험사명, 판매 시기, 출시일, 개정일 등을 **메타데이터**로 함께 저장하여 검색 정확도를 높임.  
* **Semantic Search:** 사용자가 상품명을 정확히 몰라도 유사한 키워드로 원하는 약관을 찾을 수 있도록 벡터 유사도 검색 지원.  
* **RAG (Retrieval-Augmented Generation):** 관리자가 등록한 신뢰할 수 있는 데이터 내에서만 답변을 생성하므로 할루시네이션(환각) 방지.

## **4\. 이 방식의 장점 (Why this works?)**

* **사용자 경험:** 검색 후 즉시 확인 (매우 간편)  
* **데이터 품질:** 관리자가 검증한 깨끗한 데이터(Clean Data) 기반  
* **서비스 속도:** 인덱싱된 데이터 호출로 즉각 응답  
* **법적 안전성:** 공개된 약관 정보 활용으로 개인정보 이슈 최소화

## 

## **5\. 수익 모델 및 확장성**

* **B2B SaaS:** 보험 설계사가 상담 현장에서 고객의 보험을 즉시 조회하고 보장 내용을 설명하는 전문 상담 도구로 활용 (월 구독 모델).  
* **B2C:** 일반인이 내 보험의 진짜 보장 내용을 확인하는 정보 플랫폼 (리드 생성 모델).  
* **데이터 판매:** 정교하게 구조화된 약관 데이터 자체를 다른 핀테크 기업에 API 형태로 제공.

## 

## **6\. 단계별 로드맵**

* **Phase 1 (MVP):** 주요 보험사의 베스트셀러 상품 100종에 대한 약관 DB 구축 및 벡터 검색 엔진 상용화.  
* **Phase 2:** 전체 생명/손해보험사 상품으로 데이터 범위 확장 및 메타데이터 고도화.  
* **Phase 3:** 마이데이터 API 연동을 통한 실시간 가입 내역 자동 매칭 기능 추가.

##

## **7. 주요 개발 기술적 리스크**

  ### 1. PDF 데이터 추출 및 구조화의 난이도 (가장 큰 병목)
  * 이슈: 보험 약관은 복잡한 표(Table), 계층적 목차, 각주, 예외 조항이 혼재된 PDF 형식입니다. 단순 텍스트 추출 시 문맥이 깨지거나 표 데이터가 망가질 위험이 큽니다.

  * 리스크: RAG 시스템에서 잘못된 컨텍스트를 제공하여 AI가 오답을 생성할 확률이 높아집니다.

  * 해결 방향: 단순 OCR보다는 Layout-aware PDF parsing(표 구조를 보존하는 파싱) 기술과 약관 전용 청킹(Chunking) 전략 수립이 필수적입니다.

  ### 2. 정책 버전(출시/개정일) 매칭의 정확성
  * 이슈: 동일한 상품명이라도 가입 시점(개정일)에 따라 보장 내용이 다릅니다. 사용자가 검색한 상품과 실제 약관 데이터베이스 간의 정확한 버전 매칭이 실패할 경우, 법적으로 잘못된 정보를 제공하게 됩니다.

  * 리스크: 서비스 신뢰도 하락 및 분쟁 가능성.

  * 해결 방향: 상품명 검색 시 출시일, 개정일, 가입 시기를 필수 필터로 사용하는 메타데이터 검색 아키텍처를 강화해야 합니다.

  ### 3. RAG 성능 최적화 (유사도 검색의 한계)
  * 이슈: 보험 용어는 일상어와 다르며(예: "진단비" vs "수술비"), pgvector의 단순 코사인 유사도 검색만으로는 복잡한 약관 질문에 대한 최적의 청크(Chunk)를 찾지 못할 수 있습니다.
  
  * 리스크: 질문과 관련 없는 약관 섹션을 참조하여 답변의 질 저하.
  
  * 해결 방향: Hybrid Search(키워드 기반 BM25 + 벡터 검색)와 답변 전 단계에서의 Reranking(결과 재정렬) 도입이 필요합니다.

  ### 4. 법적 책임 및 할루시네이션(환각) 방어
  * 이슈: AI가 보장 내용을 확정적으로 답변할 경우(예: "무조건 보상받습니다"), 실제 보상 거절 시 법적 책임 문제가 발생할 수 있습니다.

  * 리스크: 서비스 운영 주체의 법적 리스크.
  
  * 해결 방향: 모든 답변에 "근거가 된 약관 페이지/문구"를 시각적으로 인용(Citation)하고, "실제 보상 여부는 보험사 심사 결과에 따라 달라질 수 있다"는 강력한 면책 공지(Disclaimer) UI를 설계해야 합니다.

  ### 5. Svelte 5(Runes) 및 pgvector 성능 관리
  * 이슈: Svelte 5의 Runes 시스템은 최신 기술로, 복잡한 상태 관리(질의응답 스트리밍 등) 시 안정적인 패턴 확립이 필요합니다. 또한 데이터량이 늘어날 경우 pgvector 인덱스 최적화가 필요합니다.

  * 리스크: 대규모 데이터 로딩 시 UI 렉(Lag) 및 검색 속도 저하.
  
  * 해결 방향: Svelte 5의 $state, $derived를 활용한 효율적인 반응성 설계와 PostgreSQL 인덱스(HNSW 등) 튜닝을 병행해야 합니다.
